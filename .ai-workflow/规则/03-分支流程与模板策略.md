# 分支流程与模板策略

## 为什么拆成分支流程和文档模板

不同分支的目标、风险、文档和验收方式不同。

用一个共用大模板套所有任务，会让 AI 误判：

- Bug 修复不需要完整 PRD，但必须有复现和回归测试。
- 调研不应该直接进入开发，但必须有结论和建议。
- 重构不以新增功能为目标，而是外部行为不变。
- 新项目需要完整产品和技术设计。
- 功能迭代需要重点分析与现有系统的边界和影响。

因此，本工作流采用两层结构：

- `分支流程/`：说明某个分支要走哪些阶段、使用哪些文档、哪些必填、哪些可选。
- `文档模板/`：存放具体文档模板，按分支和文档层级组织。

## 模板拆分原则

- 不用一个模板装下需求、设计、架构、测试、验收所有内容。
- 同名模板可以存在于不同分支目录中，例如 `新项目/用户可读/需求规格.md` 和 `功能迭代/用户可读/功能需求.md`，路径决定语义。
- 只有跨分支结构高度一致的模板，才放入 `.ai-workflow/文档模板/共用/`。
- 如果一个模板里需要大量“如果是新项目填 A，如果是功能迭代填 B”，说明它应该拆分。
- 分支流程负责选择模板，具体模板负责承载内容。

## 分支类型

| 分支 | 核心问题 | 必要文档 |
| --- | --- | --- |
| 新项目 | 要做一个什么产品，为什么值得做 | 原始想法、调研、需求、产品设计、技术设计、测试验收、开发运行合约 |
| 功能迭代 | 要给现有系统增加、调整、精简或下线什么能力 | 功能需求、影响范围、设计变更、技术影响、测试验收、开发运行合约 |
| Bug 修复 | 什么坏了，如何复现，怎样证明修好 | 问题描述、复现步骤、期望行为、实际行为、根因分析、回归测试 |
| 重构 | 外部行为不变，内部改善什么 | 重构目标、现状问题、行为不变约束、影响范围、保护性测试 |
| 调研 | 要回答什么决策问题 | 背景、候选方案、对比维度、证据、风险、推荐结论、ADR |
| 发布 | 是否可以上线，如何回滚 | 发布清单、测试结果、风险确认、回滚方案、发布记录 |

## 任务档案目录

每次真实任务生成一个任务档案目录，目录写入 `.auto-spec-kit/任务档案/`：

```text
.auto-spec-kit/任务档案/
  0001-新项目-番茄钟/
    00-原始需求.md
    01-分诊结果.md
    02-调研与准备.md
    03-需求规格.md
    04-产品设计.md
    05-技术设计.md
    06-测试与验收.md
    07-开发运行合约.md
    08-开发日志.md
    09-自动评审.md
    10-人工验收.md
    11-规格回写.md
```

后续迭代、Bug、重构不要覆盖旧任务档案，应新建任务档案目录。

## 新项目和项目文档的关系

新项目仍然需要任务档案。

原因：

- 任务档案记录“这一次从想法到首版规格/开发的过程”。
- 项目文档记录“当前项目长期有效的事实”。

新项目初始化完成后，应将已确认且长期有效的内容回写到 `.auto-spec-kit/项目文档/`。

例如：

- `任务档案/0001-新项目-番茄钟/03-需求规格.md` 的确认结论，回写到 `项目文档/当前需求规格.md`。
- 技术选型结论，回写到 `项目文档/技术选型.md`。
- 架构结论，回写到 `项目文档/架构设计.md`。

## 任务档案命名规范

任务档案目录必须使用：

```text
序号-分支类型-任务名称
```

规则：

- 序号使用 4 位数字，从 `0001` 开始递增。
- 分支类型必须使用固定枚举：`新项目`、`功能迭代`、`Bug修复`、`重构`、`调研`、`发布`。
- 任务名称使用简短中文短语，不超过 20 个汉字。
- 不使用空格。

示例：

```text
0001-新项目-番茄钟
0002-功能迭代-增加白噪音
0003-Bug修复-暂停后时间跳变
0004-重构-计时状态机
0005-调研-桌面端技术选型
0006-发布-MVP上线
```

## AI 可执行模板放在哪里

不是每个分支目录都必须有 `AI可执行/`。

当前设计中，跨分支结构一致的 AI 可执行文档统一放在：

```text
.ai-workflow/文档模板/共用/AI可执行/
```

例如：

- 开发运行合约
- 开发日志
- 自动评审报告
- 规格回写记录

如果未来某个分支需要专属 AI 可执行模板，再在该分支下新增 `AI可执行/`。

## 与里程碑的关系

里程碑不是所有任务的中心。

里程碑主要用于：

- 用户想先做 MVP。
- 完整系统太大，需要分阶段交付。
- 长程 GOAL 开发需要内部检查点。
- 后续功能需要封存为未来版本。

用户最终要的是完整系统，里程碑只是交付策略。
